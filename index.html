Topic 01

OCaml Typing

Type rules, the context G, and how to construct type proofs — from your typesystem.pdf notes.

📋 Tested on Spring '24, Spring '25, Fall '23 finals
What is a Type System?

A type system is a series of rules that (1) assign types to values/expressions and (2) describe what you can do with values of a certain type. OCaml is statically and implicitly typed — type checking happens at compile time, but you don't have to write the types explicitly (type inference does it).

ℹ️
Key distinction: OCaml cannot use an int where a bool is expected (e.g., in an if-condition). Python allows this. This is because their type rules differ, not because one is "smarter."
The Context G Core Concept

Just like the environment A in Operational Semantics stores variable-to-value bindings, the context G is a partial function that maps variable names → their types.

  • G(x) = t means "variable x has type t in context G"
  • G, x : t means "extend context G with a new binding: x has type t"
  • G ⊢ e : t reads: "In context G, expression e has type t"
(T-Var)
G(x) = t
G ⊢ x : t

This says: "look up x in the context; whatever type you find is x's type."

Core Type Rules

Constant Rules (no premises needed)

(T-True)
G ⊢ true : bool
(T-False)
G ⊢ false : bool
(T-Int)
G ⊢ n : int  (for any integer n)

If Expression

(T-If)
G ⊢ e1 : bool    G ⊢ e2 : t    G ⊢ e3 : t
G ⊢ if e1 then e2 else e3 : t
⚠️
Both branches e2 and e3 must have the same type t. The condition e1 must be bool. Violating either gives a type error.

Arithmetic and Boolean Operators

(T-Add)
G ⊢ e1 : int    G ⊢ e2 : int
G ⊢ e1 + e2 : int
(T-And)
G ⊢ e1 : bool    G ⊢ e2 : bool
G ⊢ e1 && e2 : bool
(T-Eq)
G ⊢ e1 : t    G ⊢ e2 : t
G ⊢ e1 = e2 : bool

Notice T-Eq: both sides need the same type t (polymorphic), but you still can't compare an int to a bool.

Let Expression

(T-Let)
G ⊢ e1 : t1    G, x : t1 ⊢ e2 : t2
G ⊢ let x = e1 in e2 : t2

First figure out t1 (the type of e1). Then extend the context with x : t1 and check e2. The whole let expression gets e2's type.

Functions (Anonymous)

(T-Fun)
G, x : t1 ⊢ e : t2
G ⊢ fun (x : t1) → e : (t1 → t2)
(T-App)
G ⊢ e1 : (t1 → t2)    G ⊢ e2 : t1
G ⊢ e1 e2 : t2
💡
T-App is the function call rule. e1 must be a function type, and e2 must match the input type of e1. The result is the output type.
How to Derive a Type Proof Exam skill

Type proofs are like trees — you work bottom-up from the goal. Example: prove the type of let x = 3 in if true then x else x + 3

// Goal: G ⊢ let x = 3 in if true then x else x + 3 : ?

// Step 1: What type is 3? → int  (by T-Int)
G ⊢ 3 : int

// Step 2: Extend context, now check the body with G, x:int
// Need: G, x:int ⊢ if true then x else x+3 : ?

// Step 3: Check condition: G,x:int ⊢ true : bool  (by T-True)
// Step 4: Check then-branch: G,x:int ⊢ x : int  (by T-Var, G(x)=int)
// Step 5: Check else-branch: G,x:int ⊢ x+3 : int
//    → G,x:int ⊢ x : int  AND  G,x:int ⊢ 3 : int  → T-Add gives int
// Step 6: Both branches are int → T-If gives int
G, x:int ⊢ if true then x else x+3 : int

// Final by T-Let:
G ⊢ let x = 3 in if true then x else x+3 : int
🚨
Exam tip: If you cannot form a complete proof tree — the expression has a type error. Don't force it. Write "type error" and explain which rule failed.
Practice: Derive the type

What is the type of this OCaml expression? Work it out on paper, then reveal:

fun x ->
  let (a, b) = x in
  fun y ->
    let a = (a+1, b > true) in
    (a::[y])
Reveal Answer (Spring '25 Final)

Answer: (int * bool) → 'a → (int * bool) list

Breaking it down: x is destructured as a pair (a, b). a+1 forces a : int. b > true forces b : bool. Inside, a is rebound to (a+1, b > true) : int * bool. a::[y] builds a list, so y must match a's type, meaning y : int * bool. Return type is (int * bool) list.

Topic 02

Function Closures

How closures capture environments, enable partial application, and why they use static scope.

What is a Closure?

A closure is a tuple of: (1) a function definition, and (2) the environment at the time the function was created. It "closes over" the variables in scope.

let sub x y = x - y
let sub3 = sub 3
(* sub3 is a closure:
   function: (fun y -> x - y)
   environment: [x = 3]
*)
💡
A closure is not evaluated until it is called. The environment is frozen at creation time and never updated.
Partial Application & Currying Key Concept

In OCaml, every function technically takes one argument and returns either a value or another function. A "two-argument function" is really a function that returns a function:

let plus x y = x + y
(* Is syntactic sugar for: *)
let plus = fun x -> fun y -> x + y
(* type: int -> int -> int
         which is really: int -> (int -> int) *)

This lets us partially apply — supply just the first argument and get back a new function:

let add3 = plus 3
(* add3 = fun y -> 3 + y  (closure with env [x=3]) *)
add3 5  (* = 8 *)

let add10 = plus 10
add10 7  (* = 17 *)
⚠️
You can only partially apply from left to right. To partially apply the second argument, you need a helper like flip:
let flip f x y = f y x
Static Scope (Lexical Scoping)

Closures use static scope — the environment is fixed at function creation, not at call time. This means reassigning a variable after creating a closure does NOT affect the closure:

let sub x y = x - y
let x = 3
let sub3 = sub x   (* closure captures x=3 *)
let x = 5          (* rebinding x does NOT update closure *)
sub3 5             (* = 3 - 5 = -2, NOT 5 - 5 = 0 *)
ℹ️
Dynamic scope (contrast) would update the closure's variables with current top-level values, giving 0. OCaml uses static scope — the safer, more predictable model.
Partial Application Trace Exam skill
let and4 w x y z = w && x && y && z
(* = fun w -> fun x -> fun y -> fun z -> w && x && y && z *)

let and3 = and4 true
(* closure: (fun x -> fun y -> fun z -> true && x && y && z), env=[w=true] *)

let and2 = and3 true
(* closure: (fun y -> fun z -> true && true && y && z), env=[w=true,x=true] *)

let and1 = and2 true
(* closure: (fun z -> true && true && true && z), env=[w=true,x=true,y=true] *)

and1 false  (* = false *)
Topic 03

Regular Expressions

Why +, ?, and [] are syntactic sugar — and how to derive a regex from an NFA.

📋 Tested on every final — NFA/DFA conversion is always on the exam
The 5 Base Regex Operations

Everything in a regular expression can be built from just these 5 primitives:

OperationNotationMeaning
Empty stringεMatches the empty string
SymbolaMatches a single character
ConcatenationRSR followed by S
Union / AlternationR|SR or S
Kleene StarR*0 or more repetitions of R
💡
These 5 operations are sufficient to describe any regular language. All other operators are just convenient shorthand.
Why +, ?, and [ ] Are NOT Strictly Necessary Core Concept

The + Operator (one or more)

R+ means "one or more R." This is equivalent to RR* — one occurrence, followed by zero or more.

R+  ≡  RR*  ≡  R(R*)

Example: a+ ≡ aa* (matches a, aa, aaa, ...)

The ? Operator (zero or one)

R? means "zero or one R." This is equivalent to R|ε — either R or the empty string.

R?  ≡  R|ε

Example: colou?r ≡ colo(u|ε)r (matches "colour" or "color")

Character Classes [ ]

[abc] means "any one of a, b, or c." This is just union:

[abc]  ≡  a|b|c

[a-z] ≡ a|b|c|...|z (a union of 26 symbols)

⚠️
Exam answer: These operators add convenience and readability but not expressive power. Any regex using +, ?, or [] can be rewritten using only {ε, symbol, concat, |, *} to accept the exact same language.
Deriving a Regex from an NFA

To convert an NFA to a regular expression, use the state elimination method:

1
Add a new start state with an ε-transition to the original start, and a new accept state with ε-transitions from all original accept states.
2
Pick an intermediate state (not start or accept). For each pair of states (qi, qj) that pass through this state, combine the transitions into a single regex: R(qi→s) · (loop*) · R(s→qj)
3
Remove the eliminated state and add the new direct transitions between all affected pairs.
4
Repeat until only start and accept remain. The label on the final transition is your regex.
ℹ️
On exams, you're more often asked NFA→DFA (subset construction). Remember: epsilon-closures first, then build states from input transitions. Spring '25 exam had a full NFA→DFA question worth 10 pts.
Topic 04

Higher Order Functions

Map, fold_left, fold_right — implementations, traces, and how to use them.

What Makes a Function "Higher Order"?

A higher order function (HOF) is one that either takes a function as an argument or returns a function. In OCaml, functions are first-class values — they can be passed around just like integers.

(* Takes a function as input *)
let apply f x = f x

(* Returns a function *)
let get_add1 () = fun x -> x + 1
Map Must Know

Map applies a function to every element of a list, producing a new list of transformed values (same length).

Implementation

let rec map f lst = match lst with
  | []     -> []
  | h :: t -> (f h) :: (map f t)

Type signature

map : ('a -> 'b) -> 'a list -> 'b list

Full Trace: map add1 [1;2;3]

let add1 x = x + 1

map add1 [1;2;3]
= (add1 1) :: (map add1 [2;3])
= (add1 2) :: (map add1 [3])     [pushed to stack]
= (add1 3) :: (map add1 [])      [pushed to stack]
= (add1 3) :: []                  [base case]
= 4 :: []
= (add1 2) :: [4]
= 3 :: [4]
= (add1 1) :: [3;4]
= 2 :: [3;4]
= [2;3;4]
Fold Right Must Know

Fold reduces a list to a single value by combining elements right-to-left (innermost first).

Implementation

let rec fold_right f lst base = match lst with
  | []     -> base
  | h :: t -> f h (fold_right f t base)

Type signature

fold_right : ('a -> 'b -> 'b) -> 'a list -> 'b -> 'b
              ^-- element    ^-- acc   ^-- init   ^-- result

Trace: fold_right (-) [1;2;3] 0

fold_right (-) [1;2;3] 0
= (-) 1 (fold_right (-) [2;3] 0)
= (-) 1 ((-) 2 (fold_right (-) [3] 0))
= (-) 1 ((-) 2 ((-) 3 0))
= (-) 1 ((-) 2 3)       -- 3-0=3
= (-) 1 (-1)            -- 2-3=-1
= 2                     -- 1-(-1)=2
Fold Left Must Know

Fold left uses an accumulator — processes left-to-right. More memory efficient (tail recursive).

Implementation

let rec fold_left f acc lst = match lst with
  | []     -> acc
  | h :: t -> fold_left f (f acc h) t

Type signature

fold_left : ('b -> 'a -> 'b) -> 'b -> 'a list -> 'b
             ^-- acc  ^-- elem ^-- init ^-- list   ^-- result

Trace: fold_left (-) 0 [1;2;3]

fold_left (-) 0 [1;2;3]
= fold_left (-) (0-1) [2;3]   = fold_left (-) (-1) [2;3]
= fold_left (-) (-1-2) [3]    = fold_left (-) (-3) [3]
= fold_left (-) (-3-3) []     = fold_left (-) (-6) []
= -6
⚠️
fold_left vs fold_right differ for non-commutative operations!
fold_left (-) 0 [1;2;3] = -6
fold_right (-) [1;2;3] 0 = 2
They give the same result only for operations like + and * (commutative & associative).
Comparing Map vs Fold
mapfold_rightfold_left
Input'a list'a list'a list
Output'b list'b (single value)'b (single value)
Function type'a → 'b'a → 'b → 'b'b → 'a → 'b
Needs base case?No (returns [])YesYes (accumulator)
OrderLeft→RightRight→Left (inner first)Left→Right
Stack-safe?NoNoYes (tail recursive)
💡
From Spring '25: "In OCaml, anywhere we use map, we could use fold instead." — This is TRUE. You can implement map using fold_right.

In practice: Regular expressions describe tokens (lexer stage). CFGs describe sentence structure (parser stage).

Leftmost vs Rightmost Derivation

Both are ways to expand a CFG — the difference is which non-terminal you expand first.

Leftmost Derivation

Always expand the leftmost non-terminal first.

E → E + E | n

Derive: 1 + 2 + 3
E
→ E + E         (expand leftmost E)
→ E + E + E     (expand leftmost E)
→ 1 + E + E     (expand leftmost E)
→ 1 + 2 + E
→ 1 + 2 + 3

Rightmost Derivation

Always expand the rightmost non-terminal first.

E → E + E | n

Derive: 1 + 2 + 3
E
→ E + E         (expand rightmost E)
→ E + E + E     (expand rightmost E)
→ E + E + 3
→ E + 2 + 3
→ 1 + 2 + 3
ℹ️
Leftmost and rightmost derivations correspond to different parse trees for the same string when the grammar is ambiguous.
Left vs Right Recursion

Left Recursive

E → E + n | n

The non-terminal appears at the start of the production. Useful for left-associative operators (1-2-3 = (1-2)-3).

Right Recursive

E → n + E | n

The non-terminal appears at the end. Useful for right-associative operators (a::b::[] = a::(b::[])).

⚠️
Left-recursive grammars break LL parsers (like the recursive descent parsers used in class). LL parsers need to be rewritten to remove left recursion.
Ambiguity

A grammar is ambiguous if there exists a string with more than one distinct parse tree (or equivalently, more than one leftmost derivation).

(* Ambiguous grammar *)
E → E + E | E * E | n

(* "1 + 2 * 3" has two parse trees:
   (1 + 2) * 3 = 9
   1 + (2 * 3) = 7
*)

(* Unambiguous version — enforce precedence *)
E → E + T | T
T → T * F | F
F → n
🚨
Ambiguous grammars are bad for parsers — the parser doesn't know which tree to build. Fix by restructuring the grammar to reflect precedence and associativity.
Practice: Create a CFG for c*(ab)+d Spring '25 Q4

Which grammar matches the regex c*(ab)+d?

Reveal Answer

c*(ab)+d means: zero or more c's, followed by one or more "ab" pairs, followed by d.

U → M D C
M → cM | ε        (c* = zero or more c's)
D → abD | ab      (ab+ = one or more ab's)
C → dC | ε        (d, but only one d is needed → actually just d)

The correct answer from the Spring '25 exam is choice B: U → MDC, D → abD|ab, M → cM|ε, C → d|ε

(* Valid sentences: "+ 3 4"  "- 3 * 4 + 5 2" *)
Lexer Implementation

Converts a string to a list of tokens. Doesn't check grammar — just identifies valid tokens.

type token = Plus | Sub | Mult | Div | Num of int

let rec lex str =
  if str = "" then [] else
  if String.sub str 0 1 = "+" then
    Plus :: (lex (String.sub str 1 (String.length str - 1)))
  else if String.sub str 0 1 = "-" then
    Sub :: (lex (String.sub str 1 (String.length str - 1)))
  else if String.sub str 0 1 = "*" then
    Mult :: (lex (String.sub str 1 (String.length str - 1)))
  else if String.sub str 0 1 = "/" then
    Div :: (lex (String.sub str 1 (String.length str - 1)))
  else if String.sub str 0 1 = "0" then
    Num(0) :: (lex (String.sub str 1 (String.length str - 1)))
  (* ... digits 1-9 ... *)
  else lex (String.sub str 1 (String.length str - 1))

(* lex "+ 1 2" = [Plus; Num(1); Num(2)] *)
(* lex "1 2 +" = [Num(1); Num(2); Plus]  — STILL VALID for lexer! *)
(* lex "$ 1 2" = [Num(1); Num(2)]        — $ is skipped *)
ℹ️
The lexer does not check grammar. "1 2 +" lexes fine even though it's not a valid Math-ew sentence.
Parser Implementation

Takes a token list, checks grammar, and builds an AST (Abstract Syntax Tree).

type ast =
  | Int of int
  | Add of ast * ast
  | Minus of ast * ast
  | Times of ast * ast
  | Divide of ast * ast

let rec parse tok_list = match tok_list with
  | []          -> failwith "Empty"
  | [Num(x)]    -> Int(x)
  | Plus  :: Num(x) :: t -> Add(Int(x),   parse t)
  | Sub   :: Num(x) :: t -> Minus(Int(x), parse t)
  | Mult  :: Num(x) :: t -> Times(Int(x), parse t)
  | Div   :: Num(x) :: t -> Divide(Int(x), parse t)
  | _           -> failwith "Parse error"

(* parse [Plus; Num(2); Num(3)] = Add(Int(2), Int(3)) *)
(* parse [Num(1); Num(2); Plus] → FAILS — grammar violation *)
Evaluator Implementation

Traverses the AST and computes a final value via post-order traversal.

let rec eval ast = match ast with
  | Int(x)        -> x
  | Add(x, y)     -> let left = eval x in
                     let right = eval y in
                     left + right
  | Minus(x, y)   -> (eval x) - (eval y)
  | Times(x, y)   -> (eval x) * (eval y)
  | Divide(x, y)  -> (eval x) / (eval y)

(* Full pipeline: *)
eval (parse (lex "+ 2 3")) = 5
Typecheck Implementation

Checks that the AST is type-safe before evaluation (or can be run in parallel).

type typ = TInt | TBool

let rec typecheck ast ctx = match ast with
  | Int(_)        -> TInt
  | Bool(_)       -> TBool
  | Add(e1, e2)   ->
      if typecheck e1 ctx = TInt && typecheck e2 ctx = TInt
      then TInt
      else failwith "Type error: + expects int"
  | If(cond, t, e) ->
      let tc = typecheck cond ctx in
      let tt = typecheck t ctx in
      let te = typecheck e ctx in
      if tc = TBool && tt = te then tt
      else failwith "Type error in if"
  | Var(x)        -> List.assoc x ctx

Topic 07

Lambda Calculus

Complete guide from zero — syntax, β-reduction, free/bound variables, Church encodings, and the Y-combinator.

🎓
This section assumes no prior knowledge. We build up everything from scratch. Lambda calculus is the theoretical foundation of all functional languages including OCaml.
What is Lambda Calculus? From Scratch

Lambda calculus (λ-calculus) is a tiny formal system invented by Alonzo Church in the 1930s. It has only 3 constructs, yet it is Turing complete — it can compute anything a computer can compute.

Think of it as the world's smallest programming language. Modern languages like OCaml, Haskell, and Lisp are essentially elaborate extensions of lambda calculus.

The Grammar (the entire language)

E → x             (a variable)
   | λx.E         (a function definition)
   | E E           (function application)
   | (E)           (grouping)

That's it. Three constructs:

  • Variable: x, y, z — just names
  • Abstraction: λx.E — a function with parameter x and body E
  • Application: E₁ E₂ — apply function E₁ to argument E₂

Valid lambda expressions:

x               (just a variable)
λx.x            (identity function: takes x, returns x)
λx.y            (constant function: takes x, returns y)
λx.λy.x         (takes x then y, returns x — "first")
(λx.x) y        (apply identity to y — evaluates to y)
λx.λy.x y       (takes x, takes y, applies x to y)
Analogy to OCaml
Lambda CalculusOCamlMeaning

β-Reduction (How to Evaluate) Core Skill

Beta reduction is the one evaluation rule of lambda calculus. It's function application: substitute the argument into the function body.

(β-reduction)
(λx.E) V  ⇒  E[x := V]

"Replace every occurrence of x in E with V." The expression being reduced is called a redex (reducible expression).

Simple Examples

(λx.x) 5           -- identity applied to 5
⇒ 5                -- replace x with 5 in body "x"

(λx.x+1) 3        -- increment function
⇒ 3+1 = 4

(λx.λy.x) a b     -- "first" function applied to a and b
⇒ (λy.a) b        -- replace x with a in body "λy.x"
⇒ a               -- replace y with b in body "a" — y not in a, so stays a

Multi-step reduction

(λx.(λy.y x)) a b
⇒ (λy.y a) b      -- apply outer function: x → a
⇒ b a             -- apply inner function: y → b
Evaluation Order: Eager vs Lazy

Eager (Call-by-Value)

Reduce arguments before applying the function. Evaluate from inside out.

(λx.x) ((λy.y) 3)
-- First reduce argument:
⇒ (λx.x) 3
⇒ 3

Lazy (Call-by-Name)

Apply the function first, substituting the unevaluated argument. Evaluate only when needed.

(λx.x) ((λy.y) 3)
-- Apply outer first:
⇒ (λy.y) 3
⇒ 3
ℹ️
Both strategies produce the same result when they terminate (Church-Rosser theorem). But lazy evaluation can terminate when eager evaluation doesn't (e.g., if the argument causes an infinite loop but is never used). OCaml uses eager evaluation.
Free vs Bound Variables Exam Favorite

A variable is bound if it corresponds to a parameter of an enclosing lambda. It is free otherwise.

λx.x a       -- x is BOUND (it's λx's parameter in scope)
             -- a is FREE (no λa in scope)

λx.λy.x a y  -- x bound to outer λx
             -- y bound to inner λy
             -- a is FREE

(λx.λx.x) a  -- inner x is bound to inner λx (SHADOWING)
             -- outer λx's parameter is shadowed

Formal definition

FV(x)      = {x}                  -- variable: itself is free
FV(λx.E)   = FV(E) - {x}         -- abstraction: x is removed from free vars
FV(E₁ E₂)  = FV(E₁) ∪ FV(E₂)    -- application: union of free vars

BV(x)      = {}
BV(λx.E)   = {x} ∪ BV(E)
BV(E₁ E₂)  = BV(E₁) ∪ BV(E₂)
⚠️
Variable capture: When substituting, make sure you don't accidentally bind a free variable! If the substituted value has a free variable that matches a bound variable in the function body, you must rename first (alpha conversion).
α-Conversion (Alpha Equivalence) Exam Skill

Two lambda expressions are alpha equivalent if one can be obtained from the other by consistently renaming bound variables. This doesn't change the meaning.

λx.x    ≡α   λy.y    ≡α   λz.z    (all identity functions)

λx.λy.x  ≡α  λa.λb.a  ≡α  λp.λq.p  (all "first" functions)

When to use alpha conversion

-- Problematic reduction (variable capture!):
(λy.(λx.y)) x
⇒ (λx.x)           -- WRONG! Free x became bound

-- Correct: alpha-convert first to avoid clash
(λy.(λx.y)) x
→ (λy.(λa.y)) x    -- rename bound x to a
⇒ (λa.x)           -- now substitute y:=x correctly
ℹ️
Alpha conversion rule: you can rename a bound variable and its corresponding parameter consistently throughout. You cannot rename free variables.
Church Encodings Mind-bending but important

Lambda calculus has no built-in booleans, numbers, or conditionals. Church encoded these as pure lambda expressions.

Booleans

true  ≡  λx.λy.x    (takes two args, returns the FIRST)
false ≡  λx.λy.y    (takes two args, returns the SECOND)

If-then-else

Since true returns its first argument and false returns its second, we write if b then t else e as simply b t e:

if true then A else B
= (λx.λy.x) A B
⇒ (λy.A) B
⇒ A                  ✓ returns the "then" branch

if false then A else B
= (λx.λy.y) A B
⇒ (λy.y) B
⇒ B                  ✓ returns the "else" branch

Boolean operators

not   ≡ λb. b false true
and   ≡ λb.λc. b c false     (if b then c else false)
or    ≡ λb.λc. b true c      (if b then true else c)

Church Numerals (numbers as lambdas)

0 ≡ λf.λx.x           (apply f zero times)
1 ≡ λf.λx.f x         (apply f once)
2 ≡ λf.λx.f (f x)     (apply f twice)
3 ≡ λf.λx.f (f (f x)) (apply f three times)

-- Successor (n+1):
succ ≡ λn.λf.λx.f (n f x)

-- Addition:
add ≡ λm.λn.λf.λx.m f (n f x)
The Y-Combinator (Recursion) Advanced

Lambda calculus has no named functions, so how do we do recursion? Through the Y-combinator (fixpoint combinator).

Y ≡ λf.(λx.f(x x))(λx.f(x x))

Key property: Y F = F (Y F). This means Y "feeds F to itself" indefinitely:

Y F
= (λf.(λx.f(x x))(λx.f(x x))) F
⇒ (λx.F(x x))(λx.F(x x))
⇒ F((λx.F(x x))(λx.F(x x)))
⇒ F(Y F)               -- Y F = F(Y F)!
⇒ F(F(Y F))
⇒ F(F(F(Y F)))
...

Recursive factorial using Y

-- Define factorial as a non-recursive function G
-- where f is "the recursive call"
G ≡ λf.λn. if n=0 then 1 else n*(f(n-1))

-- Y G gives us the recursive factorial
(Y G) 3
= G (Y G) 3
= if 3=0 then 1 else 3*((Y G) 2)
= 3 * ((Y G) 2)
= 3 * (G (Y G) 2)
= 3 * (if 2=0 then 1 else 2*((Y G) 1))
= 3 * 2 * 1 * 1 = 6
Practice: Reduce this expression Spring '25 Q10
(λc. a b (c a)) ((λa. c) (λc. (λa. b)))

Use lazy evaluation. Order the steps:

A. (λc. a b (c a)) ((λc. (λa. b)))
B. a b ((λa. c) (λc. (λa. b)) a)
H. a b (λa. (λc. (λa. b)) a)
J. a b (λc. (λa. b))
Reveal Answer

Lazy evaluation: apply the outermost function first without reducing the argument.

Step 1: B — apply (λc.a b(ca)) to the unevaluated argument; c gets replaced by ((λa.c)(λc.(λa.b)))

Step 2: H — now reduce the application inside: (λa.c) applied to (λc.(λa.b)) gives λa.(λc.(λa.b)) a

Step 3: J — apply the inner lambda: (λa.(λc.(λa.b))) applied gives (λc.(λa.b))

Final: a b (λc.(λa.b))
